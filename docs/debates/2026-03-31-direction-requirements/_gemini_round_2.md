### [CRITICAL] 1. Claude Code UX（チャットUI）の根本的欠陥と「デスクトップアプリだからOK」の欺瞞
**引用:** 「全ModuleをClaude Code上のスキルコマンドとして統合操作する。」「/素材整理 [フォルダパス]」（提案書 3.8.2） / 「Claude Code UX: 部分同意。デスクトップアプリ（チャットUI）なので「CLI」批判は過剰。」（Opus反論）
**問題:** 動画編集という極めて視覚的・空間的なタスクを、チャットベースのUIで制御しようとするアプローチ自体が本質的に間違っています。デスクトップアプリであろうと、チャットUIである限り破綻します。
**根拠:** 動画編集者はタイムライン、映像のプレビュー、音声波形を「同時に」見て作業します。チャットに「/カット」と打ち込み、裏でXMLが生成されるのを待ち、それをPremiereにインポートして初めて結果が分かるというフローは、フィードバックループが長すぎます。結果的にPremiere上で全編見直す羽目になり、効率化どころか「AIのミスを修正する作業」という新たな苦役を生み出します。
**提案:** Claude CodeのチャットUIを完全に捨て、Premiere ProのUXP（Unified Extensibility Platform）パネルとしてネイティブな拡張機能を開発し、NLE（ノンリニア編集）のタイムライン上で直接プレビューと適用を行えるUIを構築してください。

### [CRITICAL] 2. DRM録画回避策（OBS+HDMIキャプチャ）の運用破綻と法的リスク
**引用:** 「DRM録画: 解決策あり。OBS Studio + HDMIキャプチャをオプション対応。DRMリスク明記」（Opus反論）
**問題:** ソフトウェアによる自動化システムに、物理的なハードウェア（HDMIループバックやキャプチャボード）を介在させる回避策は、無人運用において確実に破綻します。
**根拠:** Playwrightによるブラウザ操作とOBS録画を連携させる際、HDCPのハンドシェイクエラー、解像度の不一致、OSの通知ポップアップの映り込み、スリープ状態からの復帰失敗など、物理レイヤーが絡むエラーポイントが無数に発生します。これを「夜間自動スケジュール」で安定稼働させることは不可能です。さらに、HDCPの意図的な回避は著作権法上の技術的保護手段の回避にあたり、コンプライアンス上致命的です。
**提案:** 画面録画によるTVer/ABEMA等からの参考動画収集（Module 3-B）は要件から完全に削除してください。公式APIが提供されているYouTube/Vimeo、および商用ストックフッテージのAPI検索のみにスコープを限定すべきです。

### [CRITICAL] 3. ローカル/クラウド間のデータ転送アーキテクチャの矛盾と帯域枯渇
**引用:** 「rcloneでGoogle Drive→ローカルフォルダの同期を行う」「Windows Task Schedulerで毎日深夜2時に自動実行する」「4並列ダウンロード + 8並列チェックで高速化する」（提案書 3.7.2）
**問題:** 毎日深夜に大容量の動画素材を同期する設計は、ネットワーク帯域とローカルストレージを急速に食いつぶし、現場のインフラを崩壊させます。
**根拠:** 1日100GBの素材を同期すれば1ヶ月で3TBです。一般的な動画編集PCのローカルストレージは数ヶ月で枯渇します。また、n8n（クラウド）でGoogle Driveを検知し、ローカルPCへ同期し、さらにローカルでWhisperを回すなど、データがクラウドとローカルを行き来する設計は、無駄な転送時間とコストの温床です。
**提案:** クラウド（Google Drive側）で自動的に軽量なプロキシファイル（低解像度・低ビットレートH.264）を生成する仕組みを構築し、AI解析（Whisper等）やローカルへの初期同期はすべてプロキシに対して行うアーキテクチャに根本から変更してください。

### [MAJOR] 4. 「共通前処理パイプライン」による単一障害点の発生
**引用:** 「【新規】Module間の重複処理 → 共通前処理パイプライン導入を提案」（Opus反論）
**問題:** 共通前処理パイプラインの導入は、システム全体の可用性を著しく低下させる愚策です。
**根拠:** 全モジュールが単一の「共通前処理（例：FFmpeg抽出→Whisper→シーン分割）」に依存することになります。もし特定の特殊コーデックでFFmpegがクラッシュしたり、Whisperがメモリ不足（OOM）を起こした場合、素材整理、カット、テロップ生成など、後続のすべてのモジュールが連鎖的にダウンし、システム全体が停止します。
**提案:** 共通パイプライン化は避け、各モジュールが独立して必要な処理を非同期のマイクロタスクとして実行する「フォールトトレラント（耐障害性）設計」を採用してください。一部の処理が失敗しても、他の機能は動き続けるべきです。

### [MAJOR] 5. Module 1: シーン分類における「代表フレーム」の罠
**引用:** 「各シーンの代表フレーム画像をAI（Gemini Flash/Claude Vision）で分析し、背景・画角・内容を分類する」（提案書 3.1.2）
**問題:** 動画の時間的コンテキストを1枚の静止画（代表フレーム）で分析・分類することは不可能です。
**根拠:** カメラがパンしているシーン、人物がフレームインしてくるシーンなど、動画は時間変化に意味があります。「代表フレーム」がたまたまピンボケの瞬間、トランジション中、あるいは演者が目をつぶっている瞬間であれば、AIは「暗い部屋」「目閉じ」などと誤分類し、使い物にならないメタデータが量産されます。
**提案:** 代表フレーム1枚ではなく、シーン内から均等間隔で抽出した複数フレーム（例: 1シーンにつき5枚）を連結したグリッド画像をVision APIに渡すか、Gemini 1.5 Proの動画直接入力APIを使用して時間的変化を捉えさせてください。

### [MAJOR] 6. Module 2: 話者分離の非現実的な前提（声紋事前登録）
**引用:** 「出演者の声を事前登録し、未登録話者をスタッフ候補として識別する（pyannoteAI Voiceprint）」（提案書 3.2.2）
**問題:** 実際の撮影現場の音響環境において、事前登録された声紋データとのマッチングは著しく精度が落ち、誤爆が多発します。
**根拠:** 事前登録（静かな環境での録音）と、実際の現場（反響、環境音、マイクとの距離、ピンマイクとガンマイクの違い、感情による声質の変化）では音響特性が全く異なります。結果として出演者が「スタッフ（未登録）」として誤検知され、必要なシーンが無音化・カットされる致命的な事故が起きます。
**提案:** 音響的な声紋の事前登録に依存するのではなく、文字起こしされたテキストの文脈（「はいカットー」「カメラ回ってます」等のメタ発言）をLLMに判定させてスタッフの声を特定するアプローチに変更してください。
