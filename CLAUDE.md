# プロジェクトルール


## 要件定義書の管理ルール


### ファイルの扱い
- 要件定義書は `requirements.md` の1ファイルのみで管理する
- バージョンアップ時は新しいファイルを作らず、既存の `requirements.md` を直接編集する
- 絶対に `requirements_v2.md` や `requirements_v3.md` のような新ファイルを作成しないこと
- バージョン履歴はGitが自動で管理するので、ファイルを分ける必要はない
- 要件定義書のヘッダーにバージョン番号と更新日を記載して更新管理する


### 要件定義書の構成（必須）
- 要件定義書の末尾に必ず「## 実装タスク」セクションを設ける
- 実装すべき項目を `- [ ]` のチェックリスト形式で列挙する
- 完了した項目は `- [x]` に変更する
- 粒度は「1タスク = 1日〜数日で完了できる作業単位」
- Phase分けする場合は `### Phase 1` のようにサブ見出しで区切る
- このチェックリストはGitHub Issueとの自動同期に使われる


### 既に複数の要件定義書ファイルがある場合
- 最新版の内容を `requirements.md` に統合する
- 古いバージョンのファイルは `archive/` フォルダに移動する（削除はしない）
- モジュール別の要件定義書は独立ファイルとして残してOK
  - ファイル名: `requirements-[モジュール名].md`


## ファイル・フォルダの命名ルール
- ファイル名は英語のkebab-case（例: `requirements.md`, `user-guide.md`）
- 番号プレフィクス（00_, 01_）は使わない。ファイル名で内容がわかるようにする


## フォルダ構成
プロジェクトルート/
├── CLAUDE.md              # このファイル
├── requirements.md        # 要件定義書（1つだけ）
├── guide.md               # 使い方ガイド（任意）
├── src/                   # ソースコード
├── docs/                  # ドキュメント
├── archive/               # 古いバージョンの退避先
└── .github/workflows/     # GitHub Actions


## Git・GitHubのルール
- 終業時に必ずコミット＆プッシュする
- コミットメッセージは日本語で、やったことを具体的に書く
- GitHubのOrganization: BLITZ-AI-agent-team にプッシュする


## やってはいけないこと
- 要件定義書のバージョン別ファイル作成（v2, v3等）
- .env ファイルや認証情報のコミット
- node_modules/ 等の依存パッケージのコミット
